## 2026-05-08 - Component Role Branching for Screen Readers\n**Learning:** Components with both static/read-only and interactive states (like Star Ratings) often fail accessibility when using the same underlying semantic structure (e.g., disabled buttons). The screen reader experience for reading a rating vs inputting a rating requires entirely different DOM structures (img+summary vs interactive list).\n**Action:** Always split the top-level return logic based on interactivity. Use `role="img"` with a descriptive summary label for static views, and preserve interactive elements with full state mappings (`aria-pressed`, focus rings) for the input views.
